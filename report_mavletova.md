#### Вывод по 1 экперименту
Режим Exclusive обеспечивает максимальную производительность и минимальную задержку выполнения вычислений, поскольку GPU полностью выделяется одному приложению без конкуренции со стороны других workload. Такой режим оптимален для длительных и ресурсоёмких задач машинного обучения и глубокого обучения, где требуется максимальная производительность одной GPU.


## Эксперимент 2: Time-slicing
**Цель:** проверить работу GPU sharing, когда 4 pod одновременно используют одну физическую GPU.

**Конфигурация:**

```
Режим: Time-slicing
Количество pod: 4
Kubernetes resource: nvidia.com/gpu.shared
Доступный ресурс: nvidia.com/gpu.shared: 4
GPU: Tesla T4
GPU Memory: 14.56 GB
```

**Результаты benchmark:**

```
Job	avg_step_sec	min_step_sec	max_step_sec	peak_cuda_mem_gb
ts-1	4.990	      3.355	          5.488	        1.91
ts-2	4.994	      4.126	          5.485	        1.91
ts-3	4.993	      4.229	          5.475	        1.91
ts-4	4.992	      4.239	          5.477	         1.91
```

**Метрики Grafana:**

```
Peak GPU Utilization	100%
Peak Memory Copy Utilization	46%
Peak FB Used	0.00947 GB
Peak Power Usage	73.4 W
Peak GPU Temperature	71 °C
```

**Скрины из Grafana:**

**Наблюдения:**
В режиме time-slicing четыре pod одновременно выполнялись на одной физической Tesla T4. GPU utilization держался на уровне 100%, что показывает полную загрузку устройства. В сравнении с exclusive-режимом среднее время одного шага выросло примерно с 1.12 с до 4.99 с, так как GPU делилась между несколькими workload. 

**Вывод по 2 эксперименту**
Time-slicing подходит для запуска нескольких параллельных задач на одной GPU, особенно если задачи не требуют полной изоляции и максимальной производительности. Однако latency каждой отдельной job увеличивается, поэтому для одной тяжёлой задачи лучше использовать exclusive-режим.


## Эксперимент 3: MPS
MPS профиль был применён, но NVIDIA device plugin не опубликовал GPU resource (nvidia.com/gpu.shared: 0), из-за этого pod остались в состоянии Pending с причиной Insufficient nvidia.com/gpu.shared. Эксперимент MPS не был выполнен, что допустимо, так как MPS имеет экспериментальный статус.


## Итоговый вывод
В ходе лабораторной работы был развернут одновузловой GPU-кластер Kubernetes на базе Ubuntu 22.04.5 LTS и Kubernetes 1.34.8. Для управления графическим ускорителем NVIDIA Tesla T4 был установлен NVIDIA GPU Operator версии 25.10.1, который автоматически развернул драйвер NVIDIA, NVIDIA Container Toolkit, Device Plugin и DCGM Exporter.

Была выполнена проверка доступности GPU внутри Kubernetes pod с использованием утилиты nvidia-smi. Также был настроен мониторинг GPU с помощью Prometheus, Grafana и NVIDIA DCGM Exporter. В Grafana были получены метрики загрузки GPU, использования памяти, энергопотребления и температуры устройства.

В рамках практической части были исследованы различные режимы использования GPU.

В режиме Exclusive одна задача получала полный доступ к GPU. В ходе эксперимента была достигнута загрузка GPU до 100%, а среднее время выполнения шага составило около 1.12 секунды. Данный режим обеспечил максимальную производительность и минимальную задержку выполнения вычислений.

В режиме Time-slicing четыре независимые задачи одновременно использовали одну физическую GPU. Все задачи успешно выполнились, однако среднее время выполнения шага увеличилось примерно до 5 секунд из-за разделения вычислительных ресурсов между несколькими workload. При этом суммарная загрузка GPU оставалась близкой к 100%, что подтверждает эффективное использование устройства при многопользовательской нагрузке.

Также была предпринята попытка запуска режима MPS (Multi-Process Service). Конфигурация была успешно применена на уровне NVIDIA Device Plugin, однако GPU-ресурсы не были опубликованы в Kubernetes, в результате чего workload остались в состоянии Pending с ошибкой нехватки ресурса GPU. Учитывая экспериментальный статус поддержки MPS в NVIDIA Kubernetes Device Plugin, данный результат является допустимым.

По результатам проведенных экспериментов можно сделать следующие выводы:

Для одной тяжелой вычислительной задачи наиболее подходящим является режим Exclusive, поскольку он обеспечивает максимальную производительность GPU и минимальное время выполнения.
Для нескольких параллельных задач более подходящим является режим Time-slicing, который позволяет эффективно разделять одну физическую GPU между несколькими workload и повышать общую утилизацию устройства.
Режим MPS потенциально способен обеспечить более эффективное совместное использование GPU по сравнению с time-slicing, однако его поддержка зависит от конкретных версий драйверов, GPU Operator и Device Plugin.

Таким образом, все основные цели лабораторной работы были достигнуты: был развернут GPU-кластер Kubernetes, настроен мониторинг графического ускорителя, выполнены эксперименты по совместному использованию GPU и проведен анализ полученных результатов.
